# Устройство серверной части (backend): модули и поток данных

Отдельный «лёгкий» разбор **каждого модуля** crate `service_processes_core` и бинарника `service_processes`. Цель — понимать, где правила, где вход HTTP, где хранение и как это стыкуется в **гексагональной (Ports & Adapters)** архитектуре.

**Правило:** ядро (domain + application + ports) не знает про HTTP и конкретную БД. Снаружи только **адаптеры** (in-memory сейчас, позже Postgres/RabbitMQ).

---

## 1. Точка входа и режимы процесса

Файл: `backend/src/main.rs`

| Что делает | Зачем |
|------------|--------|
| Читает `APP_MODE` | `api` — HTTP-сервер; `worker` — фоновый SLA-worker |
| `build_state()` | Собирает in-memory репозитории, сервисы и состояние для Axum |
| `run_api()` | Поднимает `axum` на `0.0.0.0:8080` с маршрутами из `interfaces::http::router` |
| `run_sla_worker()` | Цикл: автоэскалация просроченных заявок + обновление кэша аналитики (`analytics_snapshot_service.refresh`) |

Итог: **один бинарник**, два **операционных** сценария развёртывания (два контейнера в k8s — тот же образ, разный `APP_MODE`).

---

## 2. Корень библиотеки

Файл: `backend/src/lib.rs`

Подключает модули:

- `domain` — сущности и правила
- `application` — сценарии использования
- `ports` — контракты «вход/выход»
- `infrastructure` — реализации портов (сейчас в основном in-memory)
- `interfaces` — адаптер входа (HTTP)

Это публичная граница пакета для тестов и возможных других бинарников.

---

## 3. Модуль `domain` — бизнес-смысл без инфраструктуры

| Файл | Роль простыми словами |
|------|----------------------|
| `domain/entities.rs` | Объекты предметной области: актив, заявка, наряд, эскалация, техник, аудит. Здесь инварианты и переходы статусов (например, допустимые смены статуса заявки). |
| `domain/value_objects.rs` | Перечисления: приоритет, статусы заявки/наряда/эскалации и т.д. Без «магии» хранения — только типобезопасность. |
| `domain/errors.rs` | Ошибки домена: не найдено, пустое поле, неверный переход, запрет по правилам (`Forbidden`). |
| `domain/analytics.rs` | DTO для **отчётов и снимков** (dashboard, SLA compliance, загрузка техников, снимок кэша). Это не «сущности учёта», а структуры для чтения/агрегации. |

Важно: домен **не** импортирует Axum, SQL, RabbitMQ.

## 4. Модуль `ports` — контракты границ

### 4.1. `ports/inbound.rs`

То, что «хочет сделать внешний мир» в терминах приложения (команды, use-case трейты). Пример: создание заявки через `ServiceRequestUseCase`.

### 4.2. `ports/outbound.rs`

То, чем приложение **пользуется** у внешнего мира:

| Группа | Назначение |
|--------|------------|
| `*Repository` | Хранение сущностей (активы, заявки, наряды, эскалации, техники, аудит). |
| `SlaPolicyPort` / `PriorityPolicyPort` | Правила расчёта SLA и приоритета по политике (можно подменить). |
| `EventPublisherPort` | Публикация событий (сейчас заглушка в консоль; позже RabbitMQ). |
| `AnalyticsQueryPort` | **Чтение и агрегации** для дашбордов: одним контрактом удобно подставить SQL/Postgres вместо загрузки всех строк в память. Методы: сводки dashboard/SLA/workload + низкоуровневые `list_*` при необходимости. |
| `AnalyticsSnapshotRepository` | Хранение **последнего снимка** KPI (обновляет worker; HTTP отдаёт из кэша). |

Порты описывают **что** нужно приложению, не **как** это сделано в PostgreSQL.

## 5. Модуль `application` — оркестрация сценариев

Каждый файл — отдельный app-сервис, который:

1. Берёт входные данные (команда или id),
2. Читает/пишет через **outbound** порты,
3. Вызывает методы **domain** для правил,
4. Пишет аудит/шлёт события по политике.

| Файл | Суть |
|------|------|
| `service_request_service.rs` | Создание заявки, смена статуса, связка с политиками SLA/приоритета и событиями. |
| `work_order_service.rs` | Наряды: создание, назначение, старт/завершение; для техника — проверка «свой наряд». |
| `escalation_service.rs` | Эскалации и разрешение. |
| `technician_service.rs` | Регистрация/список техников (как справочник исполнителей). |
| `audit_service.rs` | Запись и выборка журнала действий. |
| `sla_service.rs` | Просрочка по SLA, массовая автоэскалация (и для HTTP, и для worker). |
| `reporting_service.rs` | Тонкая обёртка над `AnalyticsQueryPort` для сценариев «отчёт» (делегирует KPI адаптеру). |
| `analytics_snapshot_service.rs` | Пересчёт полного `AnalyticsSnapshot` и чтение отдельных частей из кэша; если кэша нет — считает на лету и сохраняет. |

Здесь **нет** разбора HTTP-заголовков — этим занимается слой `interfaces`.

## 6. Модуль `infrastructure` — адаптеры реализации

Файл: `infrastructure/in_memory.rs`

| Тип | Роль |
|-----|------|
| `InMemory*` репозитории | HashMap/Arc/Mutex — для разработки и тестов; реализуют трейты `*Repository`. |
| `BasicSlaPolicy` / `KeywordPriorityPolicy` | Реализации политик SLA/приоритета. |
| `StdoutEventPublisher` | Заглушка событий (печать в консоль). |
| `InMemoryAnalyticsQuery` | Реализация `AnalyticsQueryPort`: считает агрегаты по данным в памяти (в перспективе — заменить на Postgres с тем же трейтом). |
| `InMemoryAnalyticsSnapshotRepository` | Хранит последний `AnalyticsSnapshot` в памяти процесса. |

Смена БД = новые типы в `infrastructure/…`, без переписывания домена.

---

## 7. Модуль `interfaces` — HTTP (Axum)

Файл: `interfaces/http.rs`

| Часть | Назначение |
|-------|------------|
| `AppState` | Все сервисы и репозитории, доступные обработчикам (конкретные in-memory типы в типовых ссылках — допустимый компромисс для каркаса). |
| Маршруты `/api/v1/...` | CRUD-подобные операции по активам, заявкам, нарядам, эскалациям, техникам, аудиту, дашбордам. |
| `authorize` / `parse_role` | Простая модель RBAC по заголовку `x-role` (и `x-actor-id` для техника). |
| Dashboard endpoints | Читают из **`analytics_snapshot_service`** (кэш), чтобы не пересчитывать KPI на каждый запрос при работающем worker. |
| `health` | `GET /health` → JSON `{ "status": "ok" }` для клиентов, ожидающих JSON. |

Интерфейс **только** вызывает application-сервисы и мапит ошибки в HTTP-коды.

## 8. Как проследить один запрос (пример)

**Создание заявки** `POST /api/v1/requests`:

1. HTTP-парсинг тела и заголовков (`interfaces/http.rs`).
2. Проверка роли.
3. Вызов `ServiceRequestAppService::create_request` (`application/service_request_service.rs`).
4. Внутри: проверка актива через `AssetRepository`, расчёт приоритета/SLA через порты политик, создание `ServiceRequest` в домене, сохранение в `ServiceRequestRepository`, `EventPublisherPort::publish`, при необходимости аудит.

Цепочка держит домен «внутри», а сеть и хранилище — снаружи через порты.

---

## 9. Связь с контейнерами и worker

- **API-под** (`APP_MODE=api`) обслуживает HTTP; кэш аналитики обновляется, когда клиенты вызывают dashboard и кэш пустой, или когда работает worker.
- **Worker-под** (`APP_MODE=worker`) в цикле вызывает SLA-автоэскалацию и **`analytics_snapshot_service.refresh`**, чтобы дашборды в API были «тёплыми».

Подробнее по деплою см. `infra/README.md`, контракт API — `docs/api/backend-api.md`.

## 10. Куда расширять дальше

| Направление | Где править |
|-------------|-------------|
| Новая бизнес-операция | `domain` (правила) → `ports/outbound` при новом хранилище → `application` → `interfaces/http.rs` |
| Реальная БД | Новый модуль `infrastructure/postgres_*.rs`, реализация тех же `*Repository` / расширенный `AnalyticsQueryPort` |
| Очереди | Адаптер `EventPublisherPort` на RabbitMQ в `infrastructure` |
| Аутентификация | Слой `interfaces` (JWT/API keys) + сопоставление с `x-role` или замена заголовков |

Этого достаточно, чтобы новый разработчик быстро нашёл «свой» слой и не смешивал HTTP с правилами домена.
